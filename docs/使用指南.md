# 團隊使用指南

裝完 coderigup 之後每天怎麼用。不是設計原則，是**操作手冊**。

---

## 裝完後第一次進 Claude Code

### 1. 確認安裝結果

```
.claude/
├── agents/code-reviewer.md           ← 1 個 agent
├── commands/commit.md, review.md     ← 2 個指令
├── hooks/auto-format.sh, secret-guard.sh  ← 2 個 hook
├── rules/                            ← 核心 + 偵測到的語言
├── skills/                           ← 4 個 skill
└── settings.json                     ← 權限 + hooks 掛載
CLAUDE.md                             ← 主檔（@import 各 rule）
```

### 2. 第一次進 Claude Code 做這三件事

1. **看 Claude 有沒有正確載入 context**  
   開新 session，問一句「請列出你現在看到的 rules」。應該要回你：ai-behavior、code-quality、architecture、對應語言 rule（例如 lang-go）、加上 4 個 skills。

2. **確認 hook 真的會跑**  
   叫 Claude 開一個 `.ts` 檔改一行，跑完你應該在檔案 timestamp 看到 `auto-format.sh` 跑過（prettier 整理過格式）。
   > 如果沒跑：`ls -la .claude/hooks/` 確認 `auto-format.sh` 有執行權限（`chmod +x` 若無）。

3. **測試 secret-guard**  
   叫 Claude 跑 `cat .env`。應該被擋下並印出 `BLOCKED by secret-guard`。

### 3. 看一下 `CLAUDE.md`

會看到類似這樣的 `@import`：

```markdown
- @.claude/rules/ai-behavior.md
- @.claude/rules/code-quality.md
- @.claude/rules/lang-go.md
```

這是 Claude Code 原生語法，主檔精簡但實際規則還在。不要刪掉 `<!-- coderigup:start -->` 到 `<!-- coderigup:end -->` 之間的區塊，那是 `./.coderigup-update.sh` 更新時的錨點。

## 5 步驟 flow 實際怎麼走（Claude Code）

> 以下流程預設你用的是 Claude Code。Kiro CLI 的對應做法在下一節「Kiro CLI workflow」。

| 步驟 | 你做什麼 | 工具支撐 |
|------|---------|---------|
| 1. Research | 對 Claude 描述需求 | 內建 Explore subagent 自動探 codebase |
| 2. Plan | 「改 3 個以上檔案請先列計畫」 | 內建 Plan subagent / Plan mode |
| 3. Implement | 寫 code、Claude 改檔 | `auto-format` hook 自動整理格式 |
| 4. **Verify** | **跑 `/review`** | `code-reviewer` subagent 獨立審查 |
| 5. **Commit** | **跑 `/commit`** | 自動 lint + test + 產生 message |

### 什麼時候該跑 `/review`？

- ✅ **寫完一個功能準備 commit 前** —— 最主要情境
- ✅ **重構完一段邏輯、想要獨立視角檢查** —— 主對話 context 已經被 bias，subagent 從頭看比較客觀
- ✅ **接手別人寫的 PR、要先摸清楚問題** —— 幫你列 Must Fix / Should Consider
- ❌ **每改一行就跑** —— 過度使用，code-reviewer 是「準備交件」等級的檢查
- ❌ **改 README 或純文件** —— 規則不適用，浪費 token

### 什麼時候該跑 `/commit`？

- ✅ **實際想把變更 commit 進 git 時**
- ✅ **想要標準化的 commit message 格式**
- ❌ **只是想跑 lint + test** —— 用 `npm test` / `go test` 就好，不要為了測試去跑 `/commit`
- ❌ **變更還沒準備好** —— `/commit` 會強制走完整個流程，半成品不適合

### 建議的 daily workflow

```
 開始任務
    ↓
 描述需求給 Claude（Research + Plan）
    ↓
 讓 Claude 寫 code（Implement；hook 自動 format）
    ↓
 /review           ← Verify：subagent 獨立審查
    ↓
 （修 Must Fix 項目）
    ↓
 /commit           ← 跑 lint + test + 規範化 message
    ↓
 push
```

## Kiro CLI workflow（如果你用 Kiro CLI）

Kiro CLI 跟 Claude Code 架構不一樣，**這些功能不會裝**：

- ❌ `/commit`、`/review` 這種 slash command（Kiro CLI 根本不支援自訂 slash command）
- ❌ `auto-format.sh`、`secret-guard.sh` hooks（Kiro CLI 的 hook 事件模型不同）
- ❌ 專案 `settings.json`（Kiro CLI 權限設定是機器層級，不共享進 repo）

**會裝的是**：

- ✅ `.kiro/steering/` — rules 加上 `inclusion: fileMatch` 的自動轉換
- ✅ `.kiro/steering/on-demand/` — skills 作為手動呼叫的 steering
- ✅ `.kiro/agents/code-reviewer.json` — agent 轉成 Kiro 原生 JSON 格式

### Kiro CLI 的 5 步驟實作

| 步驟 | Claude 做法 | Kiro CLI 做法 |
|------|-----------|--------------|
| 1. Research | 內建 Explore | 主對話 `kiro-cli chat` |
| 2. Plan | 內建 Plan | 主對話 |
| 3. Implement | 對話 + auto-format hook | 對話（沒 hook，要自己記得 format） |
| 4. **Verify** | `/review` | **`kiro-cli chat --agent code-reviewer "review current uncommitted changes"`** |
| 5. **Commit** | `/commit` | 手動跑 lint/test 後自己 `git commit` |

### Verify 步驟的實際指令

在你的專案目錄下，準備 commit 前：

```bash
kiro-cli chat --agent code-reviewer "review current staged + unstaged changes"
```

效果跟 Claude Code 的 `/review` 一樣 —— spawn 一個獨立 context 的 reviewer agent，依 checklist 給 Must Fix / Should Consider / Looks Good 結構化報告。

### 列出已裝的 agents

```bash
kiro-cli agent list
```

應該會看到 `code-reviewer`。

### 為什麼 hooks 不裝到 Kiro？

Kiro CLI 的 hook 事件是 `agentSpawn` / `userPromptSubmit` / `preToolUse`，跟 Claude Code 的 `PreToolUse(Bash)` / `PostToolUse(Edit|Write)` 不是直接對應。硬搬過去不但可能語意不對，還容易誤觸發。

想要等效的 `secret-guard` 行為，目前比較實際的做法是：

- 在 `kiro-cli` 的 IDE/toolsSettings 裡設 `deniedCommands` 白名單（機器層級，不會共享）
- 或接受 Kiro CLI 的這一塊能力較弱，依賴團隊個人警覺

這不是暫時的妥協 —— 是我們刻意決定「不硬搬會壞的抽象」，等 Kiro CLI hook 模型更穩定再看。

## 常見狀況排解

### secret-guard 擋住我要做的事怎麼辦？

secret-guard 的邏輯：**寧可誤擋，不要放過 leak**。誤擋的話：

1. **確認這真的是誤擋**：看 stderr 訊息，告訴你是哪個 pattern 擋的
2. **臨時放行**：改在 terminal 手動跑（不透過 Claude），不會經過 hook
3. **長期白名單**：編輯 `.claude/hooks/secret-guard.sh`，在 PATTERNS 裡拿掉或改窄對應的 regex

**不要為了方便直接關掉整個 hook**，那等於拿掉所有保護。

### auto-format 沒有跑 / 失敗

auto-format 設計上**失敗不擋**（exit 0），所以靜默失敗是刻意的。排查：

1. `which prettier` / `which gofmt` / `which ruff` —— 對應 formatter 有沒有裝？沒裝就裝一下
2. `.claude/hooks/auto-format.sh` 有沒有執行權限？`chmod +x`
3. 想看 hook 實際輸出：手動跑 `echo '{"tool_input":{"file_path":"src/foo.ts"}}' | .claude/hooks/auto-format.sh`，看有沒有錯

### hook 太慢，想暫時關掉

兩個方式：

**方法 A**（推薦）：編輯 `.claude/settings.json`，把對應的 hook 項目移到註解或刪掉。重啟 Claude Code 生效。

**方法 B**（暫時 debug 用）：把 hook script 的第一行改成 `exit 0`，等於 no-op。

**不要**直接 `rm hooks/*.sh`，下次 `./.coderigup-update.sh` 會把它們再裝回來。

### `/commit` 跑 lint 失敗

這是預期行為 —— `/commit` 強制要求乾淨。做法：

1. 看報錯、修 lint 問題
2. 再跑 `/commit`

如果你**真的**需要略過（例如 lint 壞掉是個已知別的問題），手動 `git commit` bypass 即可。`/commit` 是推薦路徑，不是強制路徑。

### 改了一個 rule 想讓它生效

`.claude/rules/*.md` 是源檔，Claude **開新 session 時才重新讀取**。改完存檔後：

1. 結束目前 session（Ctrl+C 或關視窗）
2. 重啟 `claude` 指令
3. 新 session 會套用新規則

如果改的是 `paths: frontmatter`（語言 rule 的觸發條件），可能需要完整重啟 Claude Code。

### 想加一個新 rule

先想：**這規則所有任務都需要嗎？**

- 是 → 加到 `.claude/rules/` 新檔案（沒有 `paths:` frontmatter）
- 只在特定檔案類型需要 → 加 `paths:` frontmatter
- 只在特定情境（例如寫 Dockerfile 時）需要 → 寫成 skill，放 `.claude/skills/<name>/SKILL.md`

本地新增的 rule **不會**被 `./.coderigup-update.sh` 覆蓋（update 只動 `managed-by: coderigup` 標記的檔案）。

### 想加一個新指令（`/deploy`、`/test-e2e` 之類）

**先問**：這真的值得一個指令嗎？還是直接跟 Claude 說就好？

如果確定要加：

1. 建立 `.claude/commands/<name>.md`
2. 寫 frontmatter：`description`、`argument-hint`、`allowed-tools`
3. 主體用範本式步驟描述，用 `$ARGUMENTS` 接使用者輸入

參考 `.claude/commands/commit.md` 的寫法。**但**這是**個人專案用**的擴充，**不會進團隊標準**（除非提 PR 到 coderigup repo 討論）。

## 更新到最新版

```bash
./.coderigup-update.sh
```

這會：

- 更新 `.claude/rules/`（覆寫 managed-by: coderigup 的檔案）
- 更新 `.claude/skills/`（同上）
- 更新 `.claude/agents/`、`.claude/commands/`、`.claude/hooks/`
- **不會**覆蓋現有 `.claude/settings.json`，而是放 `.claude/settings.coderigup.json` sidecar 並印出 diff 給你看
- 保留所有**非 coderigup 管理**的檔案（你自己加的 rule、skill、agent 都不會被動）

## 團隊協作建議

### Rules 怎麼演進？

1. 在**這個 coderigup repo** 提 PR
2. 團隊討論、合併
3. 所有人跑 `./.coderigup-update.sh`
4. 下次開 session 就套用新規則

**不建議**在個別專案的 `.claude/rules/` 直接改核心規則 —— 那會被下次 update 覆寫掉，而且改動沒共享給團隊。

### 誰負責維護？

一個 repo 有一個 `CODEOWNERS` 等級的負責人。核心三個 rule（ai-behavior、code-quality、architecture）改動要經過審查，語言 rule 可以相對寬鬆。

### 新人 onboarding 清單

給新加入的同事：

- [ ] 裝 Claude Code 或 Kiro CLI
- [ ] 在專案根目錄跑 `npx coderigup init`
- [ ] 讀本篇指南
- [ ] 跑一次「第一次進 Claude Code」的三個確認步驟
- [ ] 讀 `docs/逐步揭露.md` 跟 `docs/context管理.md`（了解設計原則）
