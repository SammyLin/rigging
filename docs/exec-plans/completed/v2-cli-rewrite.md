# v2 CLI Rewrite — Design Doc

Status: **Draft** — pending review
Owner: Sammy
Created: 2026-04-30

## Background

rigging currently ships via `curl | bash` running `setup.sh` (717 lines). Three upgrades are on the table:

1. Make `AGENTS.md` the primary entry point (gains coverage of Codex / Cursor / Windsurf / Amp / Devin and other open-standard tools).
2. Add a PRP template to the Plan step (borrowed from Cole Medin's context-engineering work).
3. Move distribution from curl-bash to a proper CLI tool (following GitHub Spec Kit's pattern).

Patching all three into `setup.sh` means editing bash three times and then throwing it away. Rewriting v2 directly is faster — provided existing `setup.sh` users are not disrupted.

## Goals

- **Distribution**: `npx rigging init` — zero install, instant trial.
- **Entry points**: emit `AGENTS.md`, `CLAUDE.md`, and `.kiro/steering/` from the same source of truth.
- **Methodology**: PRP template baked into the Plan step.
- **Compatibility**: `setup.sh` keeps working; v2 is developed in parallel until cutover.

## Non-goals

- Not rewriting the content of `rules/` / `skills/` / `agents/` / `commands/` / `hooks/`. This is a distribution-layer upgrade; content is reused as-is.
- Not adding tool-specific formats for Cursor / Windsurf etc. — they are covered transitively by AGENTS.md.
- Not shipping via plugin marketplaces (we want to stay tool-neutral).

## Decision Log

### D1: Language is Node + TypeScript (not Python, not Rust)

- **Why not Rust**: This CLI is I/O-bound glue. Rust's speed advantage doesn't apply, and the distribution cost (cross-platform CI matrix, signed binaries, homebrew tap) wipes out every benefit.
- **Why not Python (uv)**: Aligning with Spec Kit is appealing, but `uv` is not yet universal. Spec Kit chose Python because GitHub leans Python — that's not a reason for us to copy.
- **Why Node + TypeScript**:
    - Every Claude Code user already has `node` (Claude Code itself is built on Node).
    - `npx rigging init` works with **zero install** — a huge adoption multiplier.
    - Aligns with our own `rules/lang-node.md` (pnpm-first).
    - `npm publish` is the most well-trodden release path in this ecosystem.

### D2: CLI framework is `cac` (not `commander`)

- Lightweight, TypeScript-first, cleaner API than `commander`.
- No heavy dependency chain.

### D3: AGENTS.md is the source of truth; CLAUDE.md is a symlink

- AGENTS.md is the open standard with Linux Foundation governance.
- `CLAUDE.md` symlinks to `AGENTS.md` to avoid drift.
- Kiro reads neither symlinks nor AGENTS.md, so `.kiro/steering/` is still generated by the CLI from the source files.
- Windows lacks reliable symlinks — fall back to dual-write plus a CI consistency check.

### D4: PRP is a rule, not a skill

- PRP is a format spec for the Plan step that applies to every task → it belongs as an always-loaded rule.
- Lives at `rules/prp-template.md`, referenced from `rules/ai-behavior.md` step 2.
- Not a skill because it's not situational.

### D6: Diverge from setup.sh on three Kiro fields (P0 bugfix)

setup.sh has three Kiro output bugs the v2 port must NOT preserve. They cause the existing Kiro install to look correct on disk but behave incorrectly at runtime.

- `fileMatchPattern: "a|b|c"` (`|`-joined string) → Kiro reads it as one literal glob and matches nothing. **Fix:** emit a YAML flow array `["a", "b", "c"]`.
- `tools: ["fs_read", "execute_bash"]` → these are internal aliases, not the names Kiro accepts in `tools` / `allowedTools`. **Fix:** use `["read", "shell"]` per Kiro's [Built-in tools](https://kiro.dev/docs/cli/reference/built-in-tools/) reference.
- `toolsSettings.execute_bash.allowedCommands` → wrong outer key, the entire sandbox config is silently ignored. **Fix:** `toolsSettings.shell.allowedCommands`.

These break v2's ability to claim "byte-for-byte parity with setup.sh on `.kiro/steering/`" — but since setup.sh's output never worked correctly in Kiro, parity here would be perpetuating a bug. v2 is now the correct implementation; setup.sh should be patched separately or left to deprecate.

P1 follow-ups (DONE in same slice as P0):
- Skills moved from `.kiro/steering/on-demand/<name>.md` → `.kiro/skills/<name>/SKILL.md` (folder + SKILL.md, official Kiro layout). Agent JSON now declares `resources: ["skill://.kiro/skills/**/SKILL.md"]` so the agent can discover them.
- `allowedCommands` rewritten as anchored regex with optional-args suffix: `"git diff( .*)?"` etc. The bare `"git status"` form would only match without arguments; the `( .*)?` suffix lets each command take any args.

P2 follow-ups (still open, lower priority):
- Support the fourth inclusion mode `auto` (description-driven loading) for steering files.
- Move `managed-by: rigging` out of YAML frontmatter (non-standard key) into an HTML comment in the markdown body.

### D5: v2 ships in parallel — `setup.sh` is not deleted up front

- The new CLI lives in a separate `cli/` directory.
- `setup.sh` is frozen — only security patches.
- Once v2 is feature-complete, `setup.sh` gets a deprecation banner for one release.
- One release after that, it moves to `legacy/`. One release after that, it's removed.

## CLI Surface

```bash
# Default install (auto-detects languages)
npx rigging init

# Explicit languages
npx rigging init --lang node,python

# Show what would be written, change nothing
npx rigging init --dry-run

# Upgrade an existing install
npx rigging upgrade

# Upgrade rules only — preserve user-edited settings.json
npx rigging upgrade --rules-only

# Remove
npx rigging uninstall
```

## AGENTS.md / CLAUDE.md / Kiro Fan-out

When the CLI deploys to a target project:

1. Compose content from `rules/`, `skills/`, etc.
2. Write `AGENTS.md` as the primary file.
3. Try to symlink `CLAUDE.md` → `AGENTS.md`. On failure (Windows), fall back to dual-write and add a CI consistency check.
4. Run Kiro format conversion into `.kiro/steering/` (port the existing `setup.sh::convert_to_kiro` logic to TypeScript).
5. Copy `.claude/rules/`, `.claude/skills/`, etc. as-is.

## PRP Template Spec

`rules/prp-template.md` defines four required fields:

- **Context**: why this task exists, what upstream need drives it.
- **References**: similar prior implementations in the codebase (file path + line number).
- **Steps**: phased execution plan, each phase independently verifiable.
- **Acceptance**: how to know it's done — test list, curl examples, screenshot conditions.

Referenced from `rules/ai-behavior.md` step 2: any task touching > 3 files must produce a PRP, posted in chat or in the PR description.

## Open Questions

1. **CLAUDE.md sync on Windows**: symlink fallback — dual-write, or refuse to install on Windows? Decide based on user share.
2. **CLI self-update**: provide `npx rigging self-update`, or rely on `npx rigging@latest`? The latter is simpler and probably enough.
3. **Version pinning**: write a `.rigging-version` file into the target project's `.claude/` for upgrade diffs? Leaning yes.
4. **PRP enforcement**: hard-required for > 3 files, or always recommended? Leaning > 3 files (matches the existing `ai-behavior.md` rule).
5. **Skill distribution**: should v2 also ship in Claude Code plugin format? Leaning no — defer to v3. Land the CLI + AGENTS.md story first.

## Acceptance Criteria

All must be true before v2 cutover:

- [ ] `npx rigging init` produces a working install on clean Node, Python, and Go projects (verified by exercising the rules / agent inside both Claude Code and Kiro CLI).
- [ ] `.kiro/steering/` output is **valid per Kiro docs** — `fileMatchPattern` is an array, frontmatter uses official inclusion modes only. Note: this drops the original "byte-for-byte identical to setup.sh" criterion (see D6 — setup.sh has Kiro bugs we deliberately don't preserve).
- [ ] `.kiro/agents/code-reviewer.json` uses official Kiro tool names (`read`, `shell`) and the agent loads in `kiro-cli chat --agent code-reviewer`.
- [ ] AGENTS.md ↔ CLAUDE.md content stays in sync (symlink on Unix, dual-write merge on Windows / when CLAUDE.md pre-exists).
- [ ] PRP template and `ai-behavior.md` reference are in place.
- [ ] Both `README.md` and `README.zh-TW.md` are updated together.
- [ ] `setup.sh` shows a deprecation banner but still runs cleanly.
